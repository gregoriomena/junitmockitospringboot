4. Introducción pruebas unitarias

  • Las pruebas unitarias son un proceso de examen para verificar que una pieza de código cumple con ciertas reglas de negocio y afirmar un resultado esperado
  • JUnit es una librería java para escribir y ejecutar repetibles pruebas unitarias

5. Introducción JUnit 5

  • JUnit 4 tenía aproximadamente 10 años cuando salió la versión 5
  • Incluye programación funcional y lambda
  • JUnit 4 era monolítico, pero la versión 5 se estructura:
    • JUnit Platform: es el core. Está enfocado al contexto de ejecución del test
    • JUnit Jupiter: API con la que escribimos los tests
    • JUnit Vintage: Permite integrar la versión 4 con la 5

9. Test driven development TDD con JUnit

  1. Se desarrolla un test para un caso de aceptación de la nueva funcionalidad. Fallará porque la funcionalidad no está implementada (Rojo).
  2. Se hacen los cambios mínimos para que el test pase. (Verde)
  3. Se refactoriza
  4. Repetir
    • Se repiten estos pasos añadiendo nuevos casos hasta cubrir todos los tests de aceptación de la funcionalidad.
    • Para el ejemplo, primero creamos el test que validaba que dos instancias de Cuenta con los mismos valores las considera iguales y luego modificamos la clase Cuenta (testReferenciaCuentaWhenParamsEquals)
    • El siguiente paso fue crear el test que validaba que dos instancias de Cuenta con distintos valores las considera distintas y luego modificamos la clase Cuenta (testReferenciaCuentaWhenParamsNotEquals)
    • Luego verifica que funciona bien si el atributo "Persona" es null en alguna de las instancias a comparar (testReferenciaCuentaPersonaNull)
    • Lo mismo que el punto anterior para el atributo saldo (testReferenciaCuentaSaldoNull)
    • También se añade un test para verificar el correctofuncionamiento si el objeto a comparar no es de tipo Cuenta (testReferenciaCuentaObjetoNoInstanceOfCuenta)
    • Y por último si el objeto a comparar es null (testReferenciaCuentaNull)
